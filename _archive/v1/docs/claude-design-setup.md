# Setup in Claude Design Web

## Existing Greenfield Starter Folder

1. Create a new Claude Design project.
2. If Claude can read this starter folder, ask it to replicate the actual files from the folder.
3. Put `CLAUDE.md` at the project root/context. This is the bootstrap file.
4. Add your `DESIGN.md`, root `styles.css` facade, `starter-kit/static/tokens.css`, and the files
   inside `skills/`.
5. Add `starter-kit/static/` when you need a canvas-safe scaffold before framework handoff. Treat
   `starter-kit/static/tokens.css` as the greenfield canvas token source; generate
   `design-tokens.json` outside the canvas only when you need documentation or handoff.
6. Run the `CDP-CLAUDE-OK` canary test in `validation-method.md`. If it fails, paste
   `activation-prompt.md` as a manual fallback.
7. Start working. The protocol should guide skill selection and end deliverables, audits, final
   approval, and handoff responses with the `SKILLS APPLIED / NOT APPLIED / NEXT RECOMMENDED` block.

## Existing Brownfield Export

If the project was already generated by Claude Design, do not force the starter file layout onto it.
Many untouched exports use `colors_and_type.css` plus `preview/*.html` cards and have no root
`styles.css`, templates, or native components.

For brownfield work:

1. Read `_ds_manifest.json` first, especially `globalCssPaths`, `cards`, `tokens`, `themes`, and
   `brandFonts`.
2. Treat the existing token CSS, often `colors_and_type.css`, as the token source.
3. Keep `preview/*.html` cards as valid `@dsCard` specimens.
4. Add `styles.css` only if the user asks for normalization; then make it a facade importing the
   existing CSS, not a replacement token source.

## Empty Canvas Setup

When you cannot start from a folder, attach `CLAUDE-DESIGN-SEED.md` in the first message and use:

```text
Use CLAUDE-DESIGN-SEED.md.
Create the root CLAUDE.md first, then scaffold the static design-system structure.
Do not generate the visual design yet.
After the structure exists, ask the opening questions.
```

The first successful action should be writing the root `CLAUDE.md`. From the next turn onward, the
project is governed by that file.

After the root file exists, run `brief-framing` before visual work if the brand, audience, surface,
references, or implementation target are still unclear.

Claude Design Web does not install these files as native Skills. The `.skill.md` files are documental
skills: Claude reads them as procedures through the root `CLAUDE.md` routing layer.

Keep the lean canvas core from [`canvas-core.md`](canvas-core.md) in the project context for best
results.

Do not ask the canvas to run git, npm, Vite, Next, Astro, lint, tests, or a dev server. Keep the first
project scaffold static: HTML, CSS, and browser JS. Framework migration belongs to
[`framework-handoff.md`](framework-handoff.md).

To confirm everything is loaded and routing correctly, run the tests in
[`validation-method.md`](validation-method.md).
